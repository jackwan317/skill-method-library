## 0 为什么优秀

`knowledge-synthesis` 的优秀之处不在于调用复杂工具，而在于把“企业搜索最后一公里”定义得非常清楚：输入不是单一文档，而是来自聊天、邮件、云盘、知识库、项目管理等多源检索结果；输出也不是结果列表，而是可被用户信任的、去重后的、带来源归因的综合答案。它抓住企业知识检索的真实痛点：同一结论会散落在 Slack/Teams 线程、邮件确认、设计文档更新和任务关闭记录中，用户需要的是“到底发生了什么、依据是什么、现在还能不能信”。

该 skill 的设计强在四点。第一，先要求跨源去重，避免按来源机械罗列，迫使模型识别“同一信息在不同系统中的重复出现”。第二，把来源归因作为硬约束，要求每个结论都能追溯到 `~~chat`、`~~email`、`~~cloud storage` 等具体来源。第三，引入可信度判断，把新鲜度、权威性和多源一致性纳入答案表达。第四，按结果规模选择呈现策略：少量结果给细节，中等结果按主题聚类，大量结果给高层综合和下钻选项。这使它既适合“找一个决定”，也适合“汇总一批搜索结果”。

## 1 它要完成的任务

任务是把多源企业搜索的原始结果转换为“连贯、可信、可追溯”的答案。上游说明称它是企业搜索的 last mile：接收来自多个来源的搜索结果，执行去重、聚类、排序、可信度评估与叙事化综合，并在结尾列出来源。

它服务于 `enterprise-search` 插件整体流程：用户通过 `/enterprise-search:search` 或相关搜索工作流提出问题；系统跨连接器检索聊天、邮件、云盘、wiki、项目管理、CRM、工单等；该 skill 负责把结果变成用户可直接阅读的综合回答。它不是独立搜索器，也不负责连接 OAuth 或构造原始查询；它的边界是“已有结果之后的知识综合”。

## 2 触发条件

上游 frontmatter 标记 `user-invocable: false`，说明该 skill 不应由用户直接调用，而应由企业搜索流程在需要综合结果时自动触发。典型触发场景包括：用户询问某个决策何时做出、某项目当前状态、某主题在多个系统中的共识、谁是某领域专家、或需要从大批检索结果中提炼主要结论。

更具体地说，当搜索结果来自多个来源、存在重复信息、信息之间可能互相印证或冲突、或结果数量大到不适合逐条展示时，应使用该 skill。若只找到一条明确来源，也仍可用其归因和可信度表达规则；但若没有任何搜索结果，上游未给出专门空结果模板，需复查。

## 3 输入 / 输出

输入：多源搜索结果集合。结果通常带有来源类别占位符，例如 `~~chat`、`~~email`、`~~cloud storage`、`~~project tracker`；理想情况下还包含作者、时间、标题、频道、线程、文档名、任务状态等元数据。上游未定义严格 JSON schema，也未提供脚本接口或 API 参数格式。

输出：面向用户的自然语言答案。答案应先给结论，再给必要细节、限制与冲突说明，最后列出 sources。输出需要按规模调整：1-5 条结果时可以充分呈现每条上下文；5-15 条结果时按主题分组并列关键来源；15 条以上时提供高层综合、关键发现、顶部来源、总数和可下钻提示。

输出必须保留来源归因：直接引用可内联，完整来源列表在末尾。来源条目应包含来源类型、具体位置、日期或相对时间、相关作者、标题或线程名。聊天要写频道，邮件要写主题和发件人，云盘要写文档标题。

## 4 完成任务的步骤流程

上游给出的核心工作流是：Raw results → Deduplicate → Cluster → Rank → Assess confidence → Synthesize → Format → Coherent answer with sources。

第一步，跨源去重。判断重复的信号包括：文本相同或相近、作者/发送人相同、时间接近、引用同一项目/实体/决策、一个来源引用另一个来源。合并时使用最完整版本作为主文本，保留所有出现过的来源，并加入各来源独有细节。

第二步，聚类。不要按 `~~chat`、`~~email`、`~~cloud storage` 分栏，而应按主题、问题、决策、时间阶段或实体关系聚合。第三步，排序。把与用户问题最相关、来源最权威、时间最新、被多源支持的内容放前面。第四步，可信度评估。结合新鲜度、权威性和来源一致性，决定答案语气是确定陈述、倾向陈述还是低置信提醒。

第五步，综合成叙事。答案要像“事实链”而不是“搜索日志”：例如先说团队决定采用 REST，再说明聊天中初始讨论、邮件正式确认、设计文档已更新、项目任务已完成。第六步，格式化。根据结果量选择细节密度，并对大量结果邀请用户继续下钻。

## 5 调用的工具 / 脚本 / MCP / API

该 skill 本身没有声明可调用脚本、命令行工具或外部 API；上游 skill 目录仅发现 `SKILL.md`。它依赖企业搜索插件的上游检索能力和 MCP 连接器提供结果。插件 README 说明可搜索聊天、邮件、云盘、wiki/知识库、项目管理、CRM、工单等来源；`CONNECTORS.md` 说明 `~~category` 是工具无关占位符，会映射到实际连接的 Slack、Microsoft Teams、Microsoft 365、Dropbox、Notion、Guru、Confluence、Asana、Jira、Linear、Salesforce 等。

`.mcp.json` 中预配置或示例列出 Slack、Notion、Guru、Atlassian、Asana、Google Calendar、Gmail 等 MCP server 项。需要注意：这些是 enterprise-search 插件上下文，不是 `knowledge-synthesis` 自身的直接 API。上游未在该 skill 中描述鉴权、分页、速率限制处理或查询语法。

## 6 关键决策点

主要决策点有六类。其一，是否去重：若多个结果表达同一事实或同一决策，应合并；但同一主题有不同结论、不同观点、信息随版本演进或时间段不同，则不要合并。其二，主版本选择：重复信息中优先采用上下文最完整版本，其次权威来源，再其次最新版本。

其三，可信度语气：多源、新鲜、权威且一致时用直接陈述；单源或略旧时使用“基于……”“当时倾向于……”；陈旧、非正式或缺少正式确认时明确提示可能过时。其四，冲突处理：不能静默选择一边，必须列出冲突来源，并解释为什么较新或更权威来源更可能代表最终状态。

其五，呈现粒度：少量结果保留细节，中等结果按主题摘要，大量结果只给关键发现和顶部来源。其六，来源组织：按主题组织正文，按来源列表完成归因，避免“From chat / From email”的流水账。

## 7 验证方法 / 交付 Gate

可迁移实现时建议设置以下验证 Gate：

1. 答案是否先给结论，而不是先讲搜索过程。
2. 是否没有按来源机械罗列，而是完成跨源合并和主题聚类。
3. 每个实质性事实是否能追溯到至少一个来源。
4. 来源列表是否包含来源类型、位置、日期、作者或标题等关键元数据。
5. 重复信息是否被合并，且独有细节没有丢失。
6. 冲突信息是否显式暴露，而非被模型隐式裁决。
7. 置信度是否与新鲜度、权威性和一致性匹配。
8. 结果数量较大时是否给出摘要、支持来源数量和下钻选项。
9. 是否剔除了仅因关键词命中但与问题无关的结果。
10. 是否避免把低置信或过期信息说成确定事实。

## 8 可迁移到 Hermes Skills 的写法

最值得迁移的是“综合器提示词结构”。可以把它抽象为：输入一组带来源元数据的 evidence；先执行 duplicate detection；再按 topic/decision/entity/time cluster；再对每个 cluster 计算 relevance、authority、freshness、agreement；最后生成 answer、caveats、sources 三段。

提示词可复用以下规则：不要按来源输出，按用户问题组织；同一事实跨源出现时合并并列出所有来源；权威顺序可配置为正式知识库/最终文档/邮件公告/会议记录/聊天结论/聊天中途消息/草稿；新鲜度对状态类问题权重高，对历史事实或政策类问题权重相对低；冲突必须显式呈现；大结果集必须压缩为关键发现并提供继续追问入口。

在本地知识处理系统中，可把 `~~category` 替换成真实连接器标签，如 Slack、Gmail、Drive、Confluence、Jira；也可保留占位符以维持工具无关性。若有结构化检索结果，建议额外保留 `source_type`、`title`、`author`、`timestamp`、`url`、`snippet`、`canonical_id`、`confidence_hint` 字段，便于自动完成归因和去重。

## 9 适合归类的任务类型

候选分类：企业搜索 / 多源知识综合 / 检索后处理 / 答案合成 / 来源归因 / 可信度评估。

从 skill 类型看，它是非用户直接调用的 internal synthesis skill；从流程位置看，位于检索之后、最终回答之前；从能力看，偏“阅读理解 + 证据整合 + 风险表达”，不偏工具执行；从适配场景看，适用于企业知识库、RAG 搜索、客服知识综合、项目状态汇总、决策溯源、组织记忆整理等。

## 10 风险与注意事项

主要风险是“综合过度”。如果模型为了给出漂亮结论而合并了本应分开的观点、版本或时间段，会造成事实扭曲。第二个风险是“来源伪归因”：上游要求每个 claim 可归因，但如果输入元数据不足，模型可能生成看似完整的来源条目；迁移时应要求缺失信息写“未提供”。第三个风险是“权威误判”：不同公司中 wiki、邮件、聊天、任务状态的权威顺序可能不同，不能固定套用。

第四个风险是“时效误导”：状态类问题中，一个月前的信息可能已失效，必须提示可能过期。第五个风险是“冲突隐藏”：为了简洁而忽略早期相反意见，会让用户误解决策过程。第六个风险是隐私和权限：企业搜索结果可能包含敏感内容，skill 未讨论权限边界，需由上游 MCP 和搜索层保证只返回用户有权访问的材料。脚本、测试、示例缺失也是复用风险之一：上游未提供可执行验证样例，迁移时需自建测试集。

## 11 一句话方法论

把多源检索结果当作证据而不是列表：先去重和聚类，再按新鲜度、权威性与一致性评估可信度，最后用可追溯来源写出直接、分层、能暴露冲突的综合答案。
